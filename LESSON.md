## 1. UI и структура лендинга

> **Цель раздела:** понять, из чего состоит продающий лендинг и почему секции идут именно в таком порядке.

### Из чего состоит продающий лендинг

Вы наверняка видели десятки лендингов — страниц, которые продают один продукт или услугу. Все они построены по одной формуле. Вот **5 обязательных секций**:

| # | Секция | Задача | Что там обычно |
|:-:|--------|--------|----------------|
| 1 | **Hero** | Захватить внимание за 5 секунд | Заголовок + подзаголовок + кнопка |
| 2 | **Social Proof** | Вызвать доверие | Логотипы клиентов, цифры, отзывы |
| 3 | **Benefits** | Показать конкретную пользу | 3–6 карточек с иконками |
| 4 | **FAQ** | Снять сомнения | Вопросы-ответы в «аккордеоне» |
| 5 | **CTA** | Подтолкнуть к действию | Форма + призыв к действию |

### Почему именно этот порядок

Вы ведёте посетителя по воронке — каждая секция отвечает на вопрос, который рождается после предыдущей:

```
«Что это?»  →  «Кто этим пользуется?»  →  «Что мне это даст?»  →  «А если...?»  →  «Хочу!»
   Hero            Proof                    Benefits                FAQ              CTA
```

Или проще:

```
Внимание  →  Доверие  →  Понимание пользы  →  Снятие сомнений  →  Действие
```

> **Запомните.** Этот порядок работает для 90% лендингов. Когда будете собирать свой — начинайте с него.

### Навигация

Шапка сайта (Header) закреплена наверху экрана. В ней — ссылки на каждую секцию. Нажал «Преимущества» — страница плавно прокрутилась к нужному блоку. Это стандартный приём — **якорные ссылки**.

### Иерархия текста

Простое правило, которое помогает и посетителю, и поисковикам:

- **Один главный заголовок (H1)** — только в Hero. Это самое важное предложение на странице.
- **Заголовки секций (H2)** — по одному на каждый блок.
- **Обычный текст** — описания, пункты, ответы на вопросы.

> **Совет.** Не делайте всё одним размером шрифта — глаз должен «цепляться» за ключевые фразы.

### Промпт для AI-ассистента

Вот что можно сказать AI-ассистенту:

> *Создай лендинг на Next.js с Tailwind CSS. Нужно 5 секций: Hero с заголовком и кнопкой CTA, Social Proof с логотипами и статистикой, Benefits с 6 карточками, FAQ с аккордеоном, и CTA с формой. Навигация — фиксированная шапка с якорными ссылками на каждую секцию. Каждую секцию вынеси в отдельный компонент.*

### Что вы увидите в коде

AI создаст главную страницу примерно такой структуры:

```tsx
// src/app/page.tsx — главная страница
export default function Home() {
  return (
    <>
      <Header />     {/* Фиксированная навигация */}
      <Hero />       {/* Первый экран */}
      <Proof />      {/* Социальные доказательства */}
      <Benefits />   {/* Преимущества */}
      <FAQ />        {/* Вопросы-ответы */}
      <CTA />        {/* Форма + призыв */}
    </>
  );
}
```

Каждый компонент — отдельный файл. Это удобно: вы можете попросить AI «переделай секцию Benefits» и отправить ему только один файл, не трогая остальные.

### Чек-лист: проверьте результат

- [ ] Все 5 секций на месте
- [ ] Ссылки в шапке работают (кликните — страница прокрутится)
- [ ] Один H1 в Hero, H2 в каждой секции

> **Не работает?** Просто опишите AI, что исправить: *«Сделай заголовок в Hero крупнее»* или *«Добавь ещё один вопрос в FAQ»*.

## 2. Лид-форма и работа с данными

> **Цель раздела:** понять, что такое лид, как описать его в базе данных и как работает сохранение заявки.

### Что такое «лид»

Лид — это контакт человека, который оставил свои данные на вашем сайте. Ваша задача — собрать минимум информации, чтобы потом связаться:

| Поле | Что это | Зачем |
|------|---------|-------|
| **Имя** | Текст | Чтобы обращаться по имени |
| **Контакт** | Email или телефон | Чтобы связаться |
| **Согласие** | Галочка (да/нет) | По закону нужно явное согласие на обработку данных |

### Где хранятся лиды

Данные формы нужно куда-то сохранять. Мы используем **PostgreSQL** — это база данных, надёжное хранилище, которое не потеряет ваши заявки, даже если сервер перезагрузится.

Чтобы описать структуру данных, мы используем **Prisma** — инструмент-переводчик. Вы описываете таблицу простым языком, а Prisma создаёт её в базе:

```prisma
// prisma/schema.prisma — описание таблицы лидов
model Lead {
  id        String   @id @default(uuid())   // Уникальный ID — создаётся автоматически
  name      String                           // Имя
  contact   String                           // Email или телефон
  consent   Boolean                          // Согласие (да/нет)
  createdAt DateTime @default(now())         // Когда создан — заполняется автоматически
}
```

> **Не нужно запоминать синтаксис** — AI-ассистент сгенерирует его. Но вы должны понимать, что здесь написано, чтобы проверить: все ли поля на месте, правильные ли типы.

### Ключевые слова в модели

Обратите внимание на специальные директивы:

| Директива | Что делает | Зачем |
|-----------|-----------|-------|
| `@id` | Помечает поле как главный идентификатор | Каждую запись можно найти по этому полю |
| `@default(uuid())` | Генерирует случайный ID автоматически | Не нужно придумывать ID вручную |
| `@default(now())` | Записывает текущее время | Всегда знаете, когда создана заявка |
| `Boolean` | Строго `true` или `false` | Никаких «может быть» — для юридической однозначности |

### Защита от дублей: директива @unique

Что если один и тот же человек отправит форму дважды? В нашем стартере мы разрешаем повторные заявки — каждая получает уникальный `id`. Но в реальном проекте вы можете захотеть запретить дубли:

```prisma
contact   String   @unique   // Только одна заявка с таким email/телефоном
```

Если кто-то попытается отправить заявку с уже существующим контактом — **база данных откажет в записи**. Это надёжнее, чем проверка в коде, потому что база — последний рубеж.

> **Когда это полезно:** если вам не нужны повторные заявки.
> **Когда это мешает:** если пользователь хочет обновить свои данные. Решение зависит от бизнес-логики.

### Как это работает

Когда посетитель нажимает «Отправить»:

```
Посетитель заполняет форму и нажимает кнопку
        ↓
Браузер отправляет данные на сервер
  { name: "Иван", contact: "ivan@mail.ru", consent: true }
        ↓
Сервер проверяет: имя есть? Контакт есть? Согласие дано?
        ↓
Если всё ОК — сохраняет в базу данных
        ↓
Возвращает подтверждение: «Спасибо, заявка принята!»
```

> **Важно.** Сервер проверяет `consent` — если согласие не дано (галочка не отмечена), форма не сохранится. Это не просто UX, это юридическое требование.

### Промпт для AI-ассистента

> *Добавь форму лида в CTA-секцию. Поля: имя, контакт (email или телефон), чекбокс согласия. Форма отправляет данные на POST /api/leads. Бэкенд валидирует все поля и сохраняет через Prisma в PostgreSQL. Если согласие не дано — вернуть ошибку 400.*

### Чек-лист: проверьте результат

- [ ] Форма не отправляется без галочки согласия
- [ ] После отправки появляется сообщение «Спасибо»
- [ ] В базе данных появился новый лид (проверьте через `npx prisma studio`)

## 3. Событийность и аудит (event log)

> **Цель раздела:** научиться отслеживать действия посетителей, чтобы понимать, где теряются потенциальные клиенты.

### Зачем отслеживать события

Представьте: на ваш лендинг зашли 1000 человек, но форму заполнили только 5. Где «потерялись» остальные 995? Без аналитики вы не узнаете.

Решение — записывать ключевые действия посетителей:

| Событие | Когда срабатывает | Что показывает |
|---------|-------------------|----------------|
| `landing_view` | Страница загрузилась | Сколько людей зашли на лендинг |
| `cta_click` | Нажали кнопку «Начать» | Сколько заинтересовались |
| `lead_created` | Отправили форму | Сколько стали лидами |

### Как читать воронку

Допустим, за неделю вы видите такие цифры:

```
1000 просмотров  →  150 кликов по CTA  →  5 лидов
     (100%)            (15%)                (0.5%)
```

Теперь вы видите **узкое место**: 150 человек нажали кнопку, но только 5 заполнили форму. Значит, проблема в форме или в CTA-секции. А вот Hero работает нормально: 15% конверсия в клик — хороший показатель.

> **Ключевой вывод.** Без событий вы бы видели: «5 лидов, мало». С событиями вы точно знаете, **ГДЕ** теряете людей.

### Как это устроено

Каждое событие — строка в отдельной таблице:

```prisma
model ConversionEvent {
  id        String   @id @default(uuid())
  type      String    // "landing_view", "cta_click" или "lead_created"
  payload   Json?     // Дополнительная информация (подробности ниже)
  createdAt DateTime @default(now())
}
```

### Что такое payload и зачем он нужен

Поле `payload` — это «карман» для любой дополнительной информации. Например:

| Событие | Что можно записать в payload | Зачем |
|---------|------------------------------|-------|
| `cta_click` | `{ "section": "hero" }` | Понять, какая кнопка работает лучше |
| `cta_click` | `{ "section": "cta_form" }` | Отличить клик в шапке от клика у формы |
| `landing_view` | `{ "source": "google" }` | Узнать, откуда пришёл посетитель |

Без payload вы знаете, что было 150 кликов. С payload — что 120 из них по кнопке в Hero и 30 у формы. Это меняет ваши выводы.

### Когда события отправляются

| Событие | Где | Как |
|---------|-----|-----|
| `landing_view` | На клиенте (браузер) | Автоматически при загрузке, один раз за визит |
| `cta_click` | На клиенте (браузер) | При нажатии на кнопку CTA |
| `lead_created` | На сервере | Автоматически при сохранении лида в базу |

> **Обратите внимание.** `landing_view` отправляется один раз за сессию. Если посетитель обновит страницу — повторный просмотр не засчитается. Это даёт более честную статистику.

### Промпт для AI-ассистента

> *Добавь трекинг событий конверсии. Создай модель ConversionEvent в Prisma с полями type, payload (JSON) и createdAt. Сделай API endpoint POST /api/events. На фронтенде: отправляй landing_view при загрузке страницы (один раз за сессию), cta_click при нажатии кнопки CTA. При создании лида автоматически записывай lead_created на бэкенде.*

### Чек-лист: проверьте результат

- [ ] Откройте лендинг, затем загляните в базу (`npx prisma studio`) — должна появиться запись `landing_view`
- [ ] Нажмите кнопку CTA — в таблице событий появится `cta_click`
- [ ] Отправьте форму — появится `lead_created`

## 4. Webhook inbox

> **Цель раздела:** понять, как принимать события от внешних сервисов, защищать endpoint от подделки и не создавать дубли.

### Что такое webhook

Обычно **вы** запрашиваете данные: *«Эй, Stripe, есть новые оплаты?»* Webhook работает наоборот — **сервис сам стучится к вам**: *«У тебя новая оплата, вот данные»*.

Примеры из жизни:

| Сервис | Что присылает |
|--------|---------------|
| **Stripe** | Клиент оплатил заказ |
| **Tilda** | Посетитель заполнил форму |
| **Bitrix24** | Создана новая сделка |
| **GitHub** | Кто-то сделал коммит |

Ваша задача — создать «почтовый ящик» (endpoint), куда эти уведомления будут приходить.

### Безопасность: секретный ключ

**Проблема:** если кто-то узнает URL вашего webhook, он сможет отправлять поддельные данные. Представьте: злоумышленник шлёт фейковые «оплаты» — ваш сервис решит, что клиент заплатил.

**Решение — секретный ключ.** Отправитель добавляет секрет в заголовок запроса, а ваш сервер проверяет:

```
Правильный запрос:
  Заголовок: x-webhook-secret: мой-секрет-123
  → Сервер: "Секрет верный, принимаю данные" (201)

Поддельный запрос:
  Заголовок: x-webhook-secret: хакер-попался
  → Сервер: "Не авторизован, отклоняю" (401)
```

### Идемпотентность: защита от дублей

Слово страшное, а идея простая. Внешние сервисы часто отправляют одно и то же событие несколько раз — *«на всякий случай»*, вдруг вы не получили. Без защиты у вас появятся дубли.

**Решение:** каждое событие имеет уникальный ID. Сервер запоминает, какие ID уже обработал:

```
Первый раз:  { id: "evt-001", type: "payment" }  → "Принято, записал" (201)
Второй раз:  { id: "evt-001", type: "payment" }  → "Уже обработано"   (200)
```

> **Результат.** Сколько бы раз сервис ни отправил одно событие — в вашей базе будет **только одна** запись.

### Зачем сохранять payload

Каждое входящее событие содержит полезные данные — **payload**:

| Сервис | Событие | Что в payload |
|--------|---------|---------------|
| Stripe | Оплата | Сумма, валюта, email клиента |
| Tilda | Форма | Все поля формы |
| GitHub | Коммит | Автор, сообщение, ветка |

Мы сохраняем payload целиком. Три причины:

1. **Отладка** — если что-то пошло не так, видите, что именно прислал сервис
2. **Отложенная обработка** — можно обработать данные позже
3. **Аудит** — полная история всех входящих событий

### Модель данных

```prisma
model WebhookEvent {
  externalId  String   @unique           // ID от сервиса — для защиты от дублей
  eventType   String                     // Тип: payment, form_submit и т.д.
  payload     Json                       // Все данные — сохраняем как есть
  processedAt DateTime @default(now())   // Когда обработали
}
```

> **Обратите внимание** на `@unique` у `externalId`. Именно благодаря этому ограничению база данных не позволит создать две записи с одинаковым ID — так и работает идемпотентность.

### Промпт для AI-ассистента

> *Создай webhook inbox. Endpoint: POST /api/webhook. Требует заголовок x-webhook-secret, проверяет его по переменной окружения WEBHOOK_SECRET. Принимает JSON с полями id, event_type, payload. Сохраняет в таблицу WebhookEvent через Prisma. Поле externalId должно быть уникальным — при повторной отправке того же id возвращай 200 вместо дубликата.*

### Чек-лист: проверьте результат

- [ ] Запрос без секрета возвращает **401**
- [ ] Первый запрос возвращает **201**
- [ ] Повторный запрос с тем же `id` возвращает **200** (не 201!)

## 5. Интеграции: Telegram-уведомления

> **Цель раздела:** подключить мгновенные уведомления о новых заявках в Telegram.

### Зачем

Вы хотите узнавать о новых лидах **сразу**, а не заходить в базу данных каждый час. Telegram решает это: при каждом новом лиде — сообщение в чат.

### Как подключить

**Шаг 1. Создайте бота.**

Откройте Telegram, найдите [@BotFather](https://t.me/BotFather), напишите `/newbot`. Выберите имя и получите токен — строку вида `123456:ABC-DEF...`.

**Шаг 2. Узнайте ID чата.**

Добавьте бота в нужный чат или группу. Отправьте любое сообщение. Затем откройте в браузере:

```
https://api.telegram.org/bot<ВАШ_ТОКЕН>/getUpdates
```

В ответе найдите число в поле `chat.id` — это ID чата.

**Шаг 3. Сохраните в переменных окружения.**

```env
TG_BOT_TOKEN=123456:ABC-DEF...
TG_CHAT_ID=-1001234567890
```

### Что получите

Когда кто-то оставляет заявку, вам мгновенно приходит:

```
Новая заявка!

Имя: Иван Петров
Контакт: ivan@mail.ru
ID: 550e8400-...
Время: 2026-04-01T19:22:18Z
```

### Важный принцип: fire-and-forget

> **Уведомление — бонус, а не основная функция.**

Если Telegram временно недоступен — лид **всё равно сохранится** в базе данных. Мы не ждём ответа от Telegram: отправили и забыли. Данные важнее уведомления.

Представьте аналогию: вы записали заказ в блокнот (база данных) и отправили SMS коллеге (Telegram). Если SMS не дошло — заказ в блокноте остался. Потерять уведомление — неприятно. Потерять лид — критично.

### Промпт для AI-ассистента

> *Добавь Telegram-уведомления. Создай функцию sendTelegramMessage в src/lib/telegram.ts — она отправляет сообщение через Telegram Bot API (sendMessage, parse_mode: HTML). В route POST /api/leads после создания лида вызывай эту функцию fire-and-forget (не блокируй ответ клиенту). Токен и chat_id бери из переменных окружения TG_BOT_TOKEN и TG_CHAT_ID.*

## 6. Деплой и инфраструктура

> **Цель раздела:** вывести приложение в интернет — чтобы к нему мог зайти любой человек по ссылке.

### Зачем деплоить

Пока ваш лендинг работает только на вашем компьютере. Чтобы его увидели другие — нужно разместить его на сервере и получить URL.

### Что нужно для деплоя

| Инструмент | Что делает | Аналогия |
|------------|-----------|----------|
| **Сервер (VPS)** или **Vercel** | Машина в облаке, где будет жить ваш сайт | Квартира, в которую вы переезжаете |
| **Docker** | Упаковывает приложение со всеми зависимостями | Коробка для переезда — внутри всё, что нужно |
| **docker-compose** | Запускает несколько контейнеров одной командой | Грузовик, который привозит все коробки сразу |

### Переменные окружения

Секреты (пароли, токены) **никогда** не хранятся в коде. Они лежат в файле `.env`, который не попадает в Git:

| Переменная | Что это | Где узнать |
|------------|---------|-----------|
| `DATABASE_URL` | Адрес базы данных | При создании БД |
| `TG_BOT_TOKEN` | Токен Telegram-бота | У @BotFather |
| `TG_CHAT_ID` | ID чата для уведомлений | Через getUpdates |
| `WEBHOOK_SECRET` | Секрет для вебхуков | Придумайте сами |

> **Правило.** Если значение — **секрет** или **меняется** между вашим компьютером и сервером — оно идёт в `.env`. Всё остальное — в коде.

### Как деплоить

```bash
# 1. Клонируем проект на сервер
git clone https://github.com/jklazer/vibe-landing-starter.git
cd vibe-landing-starter

# 2. Создаём файл с секретами
cp .env.example .env
nano .env  # заполняем реальные значения

# 3. Запускаем всё одной командой
docker compose up -d --build

# 4. Проверяем
curl http://localhost:3000
```

### Что при этом происходит

```
docker compose up -d --build
        ↓
Docker собирает образ приложения (по инструкциям из Dockerfile)
        ↓
Поднимает контейнер с PostgreSQL
        ↓
Ждёт, пока база данных станет доступна
        ↓
Запускает контейнер с приложением
        ↓
Приложение применяет миграции (создаёт таблицы) и стартует
        ↓
Лендинг доступен по URL
```

### Volume: чтобы данные не пропали

В `docker-compose.yml` есть настройка **volume** — она хранит данные базы **вне контейнера**. Даже если вы перезапустите или пересоздадите контейнер — все ваши лиды и события останутся. Без volume каждый перезапуск обнулял бы базу.

### Промпт для AI-ассистента

> *Настрой деплой. Создай Dockerfile с multi-stage build для Next.js (standalone output). Создай docker-compose.yml с двумя сервисами: app (наше приложение) и db (PostgreSQL 16). Добавь healthcheck для базы, volume для данных, env_file для переменных. Приложение должно ждать готовности БД перед стартом и автоматически применять миграции.*
